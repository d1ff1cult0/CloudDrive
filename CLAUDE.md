# Vault — architecture & contributor notes

Self-hosted cloud storage (Google Drive style). Behind a Cloudflare Tunnel on
port **3033**. Multi-user, multi-backend (local drives + S3), whitelist signup,
quotas, per-folder backend targeting, public share links.

## Golden rules (do not break)

1. **Never buffer file payloads in RAM.** All file I/O streams: request body →
   driver, driver → response body. Any `req.arrayBuffer()` / `Buffer.concat` on a
   file payload is a bug. (Small control payloads/JSON are fine.)
2. **The app never touches the filesystem or S3 directly.** All physical I/O goes
   through the `StorageDriver` interface (`src/lib/storage/`). Routes/services are
   storage-agnostic.
3. **Cloudflare limits are hard:** no request body > 100 MB (we use 50 MiB
   chunks), no response > ~100 s (we use HTTP Range so clients re-request windows).
4. Upload/download route handlers declare `export const runtime = "nodejs"`.
5. TypeScript strict. Zod-validate every API input. Use Prisma transactions for
   anything touching quotas or upload state.
6. Path safety: user names are sanitized; physical storage keys are
   server-generated (`<ownerId>/<fileId>`, UUID-based), never user-derived. The
   local driver rejects any resolved path escaping its `basePath`.
7. **Quota accounting only changes inside the same transaction that flips
   `File.status`** (READY on complete, decrement on delete).

## Toolchain gotchas (newer than typical training data)

- **Prisma 7**: generator is `prisma-client` (output `src/generated/prisma`, gitignored,
  regenerated by `postinstall`). No `url = env()` in `schema.prisma` — the datasource
  URL lives in `prisma.config.ts` (which also holds `migrations.seed`). **Requires a
  driver adapter**: the singleton `src/lib/db.ts` uses `@prisma/adapter-pg`. Import the
  client/enums/model-types from `@/generated/prisma/client` (relative in files that tsx
  runs, e.g. the seed). Engine-free (WASM query compiler; no native binaries).
- **Next.js 16**: route `params` is a **Promise** (`await params`). The middleware
  convention was renamed to **`proxy`** — see `src/proxy.ts` (`export function proxy`).
  Build uses Turbopack and bundles server deps into chunks. `output: "standalone"`.
- **BetterAuth**: server instance `src/lib/auth.ts` (prisma adapter, email+password,
  `nextCookies()` plugin last). Sign-out needs `Content-Type: application/json` + `{}`.
- **Zod 4**: use `z.url()`, `z.uuid()`, `z.email()` (top-level), `z.coerce.bigint()`.
- **pnpm** is pinned via `packageManager` so Docker's corepack matches the host and
  the lockfile format is stable.

## Auth model

- Session gate in `src/proxy.ts` (edge, `getSessionCookie` — no DB): `/api/*` → 401
  when unauthenticated; app pages → redirect `/login`. Public: `/api/auth`,
  `/api/share/` (trailing slash!), `/s/`. Real authorization is re-checked in
  handlers via `getCurrentUser()` / `requireAdmin()`.
- **First registrant becomes admin**: `databaseHooks.user.create` in `auth.ts`
  lets the first user through (no whitelist) and promotes them to ADMIN; everyone
  after must be in `WhitelistedEmail`.

## Storage / backends

- `getDriverForBackend(id)` builds & caches a driver from the `StorageBackend` row;
  `resolveBackendForFolder(folderId)` walks up the tree to the nearest
  `storageBackendId`, else the default. Exactly-one-default is enforced in
  `admin-service.ts`. A file's backend is frozen at upload init.

## Running & verifying

- Dev: run a local Postgres (the compose one isn't host-published) —
  `docker run -d --name vault-pg -p 5432:5432 -e POSTGRES_USER=vault -e POSTGRES_PASSWORD=vault -e POSTGRES_DB=vault postgres:16`
  then `pnpm prisma migrate dev && pnpm prisma db seed && pnpm dev`.
- Tests: `pnpm test` (vitest — local driver round-trip + path-escape; Range parser).
- Full stack: `docker compose up -d` (postgres + one-shot `migrate` + `app` on 3033).
- After changes, run `pnpm build` and `pnpm prisma validate`.
- Manual API checks: sign in via `POST /api/auth/sign-in/email` (send
  `Origin: $APP_URL`), reuse the cookie jar. Upload = init → PUT chunk(s) →
  complete. Download supports `Range`/`HEAD`.

## Deploy (Docker)

Multi-stage `Dockerfile`: `deps` → `builder` (standalone) → slim `runner`
(non-root via `su-exec`, entrypoint auto-generates `BETTER_AUTH_SECRET`) and a
`migrator` target (full toolchain, runs `migrate deploy` + `db seed`). Compose
wires postgres + one-shot migrate + app; the only operator setting is
`STORAGE_PATH` (host dir → `/data/storage`, the default backend's basePath).
